# Phase 7: Update UI - Research

**Researched:** 2026-03-09
**Domain:** Tauri v2 updater frontend integration, React update notification UX
**Confidence:** HIGH

## Summary

Phase 7 builds the user-facing update experience on top of the infrastructure established in Phases 5 and 6. The Rust backend already has `tauri-plugin-updater` (Builder pattern) and `tauri-plugin-process` registered, the `tauri.conf.json` is fully configured with the Ed25519 public key, GitHub Releases endpoint, and passive NSIS install mode, and the CI pipeline produces signed `latest.json` manifests. This phase is purely frontend: check for updates on app launch, display a notification banner, show release notes, handle download with progress, and display the current app version.

The `@tauri-apps/plugin-updater` JS package (already installed at `^2.10.0`) provides `check()` which returns an `Update` object containing `version`, `body` (release notes from `latest.json` "notes" field), and `downloadAndInstall(onEvent)` with progress callbacks. The `@tauri-apps/plugin-process` package (already installed at `^2.3.1`) provides `relaunch()`. The `@tauri-apps/api/app` module provides `getVersion()` for displaying the current version. No new npm or Cargo dependencies are needed.

Critical Windows behavior: on Windows, the NSIS installer automatically exits the app when the install step executes. This means after `downloadAndInstall()` resolves, the app is already closing. The `relaunch()` call handles restart. Users should be warned that clicking "Install" will briefly close the app.

**Primary recommendation:** Create a `useUpdateChecker` hook that calls `check()` on mount, an `UpdateBanner` component rendered in `App.tsx` between the header and main content (following the existing `IntegrityAlert` pattern), and add a version display to the header or settings panel. Use Zustand for update state since the update lifecycle spans multiple user interactions.

### Pattern 1: Update State Machine (Zustand Store)

**What:** A Zustand store managing the update lifecycle as a state machine
**When to use:** When state spans multiple components and user interactions

```typescript
// Source: project pattern from src/store/gameStore.ts + Tauri updater API
import { create } from 'zustand'
import type { Update } from '@tauri-apps/plugin-updater'

type UpdateStatus = 'idle' | 'checking' | 'available' | 'downloading' | 'error'

interface UpdateState {
  status: UpdateStatus
  update: Update | null
  progress: number          // 0-100
  contentLength: number     // total bytes
  downloaded: number        // bytes downloaded so far
  error: string | null

  setChecking: () => void
  setAvailable: (update: Update) => void
  setDownloading: () => void
  setProgress: (chunkLength: number) => void
  setContentLength: (len: number) => void
  setError: (msg: string) => void
  reset: () => void
}
```

### Pattern 2: Non-blocking Update Check Hook

**What:** A hook that checks for updates on mount without blocking UI
**When to use:** Called once in App.tsx on launch

```typescript
// Source: Tauri v2 updater docs + project pattern from useIntegrityScan.ts
import { useEffect } from 'react'
import { check } from '@tauri-apps/plugin-updater'
import { useUpdateStore } from '../store/updateStore'

export function useUpdateChecker() {
  const { setChecking, setAvailable, setError, status } = useUpdateStore()

  useEffect(() => {
    if (status !== 'idle') return

    let cancelled = false
    setChecking()

    check()
      .then((update) => {
        if (cancelled) return
        if (update) {
          setAvailable(update)
        } else {
          // No update available -- reset to idle
          useUpdateStore.getState().reset()
        }
      })
      .catch((err) => {
        if (cancelled) return
        // Silently fail -- update check should never block the app
        console.error('Update check failed:', err)
        setError(String(err))
      })

    return () => { cancelled = true }
  }, [])
}
```

### Pattern 3: Download with Progress Tracking

**What:** Call `downloadAndInstall()` with event callback to track progress
**When to use:** When user clicks "Install" button

```typescript
// Source: https://v2.tauri.app/plugin/updater/
import { relaunch } from '@tauri-apps/plugin-process'

async function handleInstall(update: Update) {
  const store = useUpdateStore.getState()
  store.setDownloading()

  await update.downloadAndInstall((event) => {
    switch (event.event) {
      case 'Started':
        store.setContentLength(event.data.contentLength ?? 0)
        break
      case 'Progress':
        store.setProgress(event.data.chunkLength)
        break
      case 'Finished':
        break
    }
  })

  // On Windows, the app is already exiting at this point due to NSIS installer.
  // relaunch() ensures the app restarts after the installer finishes.
  await relaunch()
}
```

### Anti-Patterns to Avoid
- **Blocking the UI during update check:** `check()` must run in background. Never `await` it in render path or show a loading spinner for it
- **Using React Query for update state:** The update object has methods (`downloadAndInstall`, `install`). This is imperative state, not server-fetched data. Zustand fits better
- **Calling relaunch() without downloadAndInstall():** On Windows, the installer handles the relaunch. But `relaunch()` is still needed as a fallback and works cross-platform
- **Forgetting to handle check() errors:** Network failures, DNS issues, offline mode. Must catch silently -- never show error for failed update check
- **Large release notes without scroll:** Release notes from `generateReleaseNotes` can be lengthy. Use a collapsible section or dialog with overflow-auto

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Update checking | Custom fetch to latest.json | `check()` from `@tauri-apps/plugin-updater` | Handles signature verification, version comparison, platform matching |
| Download + install | Custom download with fetch API | `update.downloadAndInstall()` | Handles atomic install, signature verification, platform-specific installer invocation |
| App version | Read package.json or config | `getVersion()` from `@tauri-apps/api/app` | Reads the authoritative version from tauri.conf.json at runtime |
| Progress calculation | Manual byte tracking | Accumulate `chunkLength` from Progress events against `contentLength` from Started event | The callback provides all needed data |

**Key insight:** The `@tauri-apps/plugin-updater` API is designed as the complete frontend interface. `check()` returns a rich `Update` object with version, body (release notes), date, and action methods. No Rust command needs to be written for this phase.

## Common Pitfalls

### Pitfall 1: Update Object Lifetime
**What goes wrong:** The `Update` object returned by `check()` extends `Resource` and must be cleaned up. If the user dismisses the banner and never installs, the resource leaks.
**Why it happens:** `Update` holds a resource ID (`rid`) that must be freed.
**How to avoid:** Call `update.close()` when the user dismisses the notification without installing. Store the `Update` object in Zustand and call `close()` in the reset/dismiss action.
**Warning signs:** Memory warnings in dev tools; `Update` objects accumulating.

### Pitfall 2: Windows Auto-Exit on Install
**What goes wrong:** On Windows, calling `downloadAndInstall()` triggers the NSIS installer which automatically exits the app. Code after `downloadAndInstall()` may or may not execute.
**Why it happens:** Windows NSIS installers need to replace the running executable, so they terminate the process.
**How to avoid:** Show a clear message before install: "The app will briefly close to install the update." Don't rely on code running after `downloadAndInstall()` resolves on Windows.
**Warning signs:** App closes abruptly without user expectation.

### Pitfall 3: contentLength May Be Undefined
**What goes wrong:** The `Started` event's `contentLength` can be `null`/`undefined` if the server doesn't send `Content-Length` header (e.g., chunked transfer).
**Why it happens:** GitHub Releases typically does send Content-Length, but edge cases exist (CDN, redirects).
**How to avoid:** Handle `contentLength === 0` or `undefined` gracefully -- show an indeterminate progress bar instead of percentage-based.
**Warning signs:** Progress bar shows NaN% or 0%.

### Pitfall 4: Checking for Updates in Dev Mode
**What goes wrong:** `check()` fails in development because there's no valid `latest.json` at the endpoint matching the dev version.
**Why it happens:** The updater compares versions and the dev build version (0.1.0) has no corresponding release.
**How to avoid:** Wrap the update check in a try/catch. In dev mode, the error is expected and should be silently ignored. Consider checking `import.meta.env.DEV` to skip entirely in dev mode.
**Warning signs:** Console errors on every dev launch.

### Pitfall 5: Race Condition with Multiple Check Calls
**What goes wrong:** If the update check hook runs twice (React StrictMode double-mount), two `Update` objects could be created.
**Why it happens:** React 18+ StrictMode mounts/unmounts/remounts in development.
**How to avoid:** Use a status guard (`if (status !== 'idle') return`) in the hook and cleanup function that sets `cancelled = true`.
**Warning signs:** Duplicate update banners; resource leaks in dev mode.

## Code Examples

### Complete Update Store
```typescript
// src/store/updateStore.ts
// Source: Zustand pattern from src/store/gameStore.ts + Tauri updater API
import { create } from 'zustand'
import type { Update } from '@tauri-apps/plugin-updater'

type UpdateStatus = 'idle' | 'checking' | 'available' | 'downloading' | 'installing' | 'error'

interface UpdateStore {
  status: UpdateStatus
  update: Update | null
  downloaded: number
  contentLength: number
  error: string | null

  setChecking: () => void
  setAvailable: (update: Update) => void
  setDownloading: () => void
  addProgress: (chunkLength: number) => void
  setContentLength: (len: number) => void
  setInstalling: () => void
  setError: (msg: string) => void
  dismiss: () => void
}

export const useUpdateStore = create<UpdateStore>((set, get) => ({
  status: 'idle',
  update: null,
  downloaded: 0,
  contentLength: 0,
  error: null,

  setChecking: () => set({ status: 'checking' }),
  setAvailable: (update) => set({ status: 'available', update }),
  setDownloading: () => set({ status: 'downloading', downloaded: 0, contentLength: 0 }),
  addProgress: (chunkLength) => set((s) => ({ downloaded: s.downloaded + chunkLength })),
  setContentLength: (len) => set({ contentLength: len }),
  setInstalling: () => set({ status: 'installing' }),
  setError: (msg) => set({ status: 'error', error: msg }),
  dismiss: () => {
    const { update } = get()
    if (update) update.close()
    set({ status: 'idle', update: null, downloaded: 0, contentLength: 0, error: null })
  },
}))
```

### Complete Update Check Hook
```typescript
// src/hooks/useUpdateChecker.ts
import { useEffect } from 'react'
import { check } from '@tauri-apps/plugin-updater'
import { useUpdateStore } from '../store/updateStore'

export function useUpdateChecker() {
  const status = useUpdateStore((s) => s.status)
  const setChecking = useUpdateStore((s) => s.setChecking)
  const setAvailable = useUpdateStore((s) => s.setAvailable)
  const setError = useUpdateStore((s) => s.setError)
  const dismiss = useUpdateStore((s) => s.dismiss)

  useEffect(() => {
    if (status !== 'idle') return

    let cancelled = false
    setChecking()

    check()
      .then((update) => {
        if (cancelled) return
        if (update) {
          setAvailable(update)
        } else {
          dismiss()
        }
      })
      .catch((err) => {
        if (cancelled) return
        console.error('Update check failed:', err)
        // Silently reset -- never block the app for a failed update check
        dismiss()
      })

    return () => { cancelled = true }
  }, [])
}
```

### Complete Banner Component Structure
```typescript
// src/components/UpdateBanner.tsx
// Pattern follows IntegrityAlert.tsx exactly
import { Download, X, ChevronDown, ChevronUp } from 'lucide-react'
import { Button } from '@/components/ui/button'
import { useUpdateStore } from '../store/updateStore'
import { relaunch } from '@tauri-apps/plugin-process'

export default function UpdateBanner() {
  const { status, update, downloaded, contentLength, dismiss } = useUpdateStore()

  if (status !== 'available' && status !== 'downloading' && status !== 'installing') {
    return null
  }

  const progress = contentLength > 0 ? Math.round((downloaded / contentLength) * 100) : 0

  async function handleInstall() {
    if (!update) return
    const store = useUpdateStore.getState()
    store.setDownloading()

    try {
      await update.downloadAndInstall((event) => {
        switch (event.event) {
          case 'Started':
            store.setContentLength(event.data.contentLength ?? 0)
            break
          case 'Progress':
            store.addProgress(event.data.chunkLength)
            break
          case 'Finished':
            store.setInstalling()
            break
        }
      })
      await relaunch()
    } catch (err) {
      store.setError(String(err))
    }
  }

  return (
    <div className="mx-4 mt-3 flex items-start gap-3 rounded-md border border-blue-500/40 bg-blue-500/10 p-3 text-sm">
      <Download className="h-4 w-4 text-blue-400 mt-0.5 shrink-0" />
      <div className="flex-1">
        {status === 'available' && (
          <p className="text-blue-200">
            Version {update?.version} is available.{' '}
            <button onClick={handleInstall} className="underline font-medium">
              Install now
            </button>
          </p>
        )}
        {status === 'downloading' && (
          <div>
            <p className="text-blue-200 mb-1">Downloading update...</p>
            <div className="h-1.5 rounded-full bg-blue-500/20 overflow-hidden">
              <div
                className="h-full bg-blue-500 transition-all"
                style={{ width: contentLength > 0 ? `${progress}%` : '100%' }}
              />
            </div>
          </div>
        )}
        {status === 'installing' && (
          <p className="text-blue-200">Installing update... The app will restart.</p>
        )}
      </div>
      {status === 'available' && (
        <Button
          variant="ghost" size="icon"
          className="h-6 w-6 shrink-0 text-muted-foreground"
          onClick={dismiss}
          aria-label="Dismiss update notification"
        >
          <X className="h-3.5 w-3.5" />
        </Button>
      )}
    </div>
  )
}
```

## Open Questions

1. **Release notes rendering (Markdown vs plain text)**
   - What we know: `update.body` contains release notes. With `generateReleaseNotes: true`, GitHub generates Markdown-formatted notes
   - What's unclear: Whether to render Markdown or show as plain text
   - Recommendation: Show as plain text in a collapsible `<pre>` block. Adding a Markdown renderer is scope creep for a mod manager. The notes are readable as-is.

2. **Update check timeout**
   - What we know: `check()` accepts an optional `timeout` in CheckOptions. Default is unknown (likely 30s)
   - What's unclear: What timeout is appropriate for a background check
   - Recommendation: Use a 10-second timeout. Users on slow connections won't be blocked, and 10s is enough for GitHub's CDN. Pass `{ timeout: 10000 }` to `check()`.
